1. The Problem We Are Solving


1.1  What Exists Today in SensAI (The Baseline)
SensAI already has a working, production-grade LMS. Mentors can see three things:

•	Completion percentages per cohort via GET /cohorts/{id}/courses/{course_id}/metrics
•	A leaderboard sorted by streak and tasks completed via GET /cohorts/{id}/leaderboard
•	Raw chat history per learner per question stored in the chat_history table

That is a reporting tool. It answers one question: what happened.

A mentor with 40 learners opens the dashboard and sees:
Arjun: 34% complete
Priya: 67% complete
Ravi:  12% complete
Now what? Who do they call first? Why is Ravi at 12%? Is it because he is lazy, confused, overwhelmed, or did he just join late? The dashboard has no answer. The mentor guesses, or ignores it.

The creator has it worse. They publish a task, it gets low completion, and they have zero signal on whether the content is bad or the learners are disengaged. They regenerate blindly.

1.2  The Real Problem

Persona	The Problem
Mentors	Drowning in data but starving for decisions
Creators	Flying blind on whether their content works
The Gap	Signal extraction and action routing are missing

The gap is not data collection. SensAI already collects everything. The gap is connecting chat_history + task_completions + scorecards into a decision layer that tells you:

•  "Arjun is individually struggling with recursion — mentor should intervene."
•  "Module 3 is broken for 40% of the cohort — creator must fix the content."
These are two completely different problems requiring two completely different actions. No existing tool in the LMS space makes that distinction automatically.

1.3  Why No Other Team Solves This

Most teams will build a completion dashboard with colored cards. Some will add a basic alert for low-completion learners. None will build the Individual vs Systemic classifier — the engine that determines whether a struggling learner’s problem is unique to them or is actually a broken content module affecting 40% of the cohort.

That distinction is the entire product.

And on top of that distinction, this system does something no rule-based system ever will: it learns. The signal weights that determine who is truly struggling are not hard-coded guesses. They adapt per cohort using a Contextual Multi-Armed Bandit, validated by real intervention outcomes.

 
2. Why Reinforcement Learning Instead of Fixed Rules


2.1  The Problem with Fixed Weights
The standard approach to friction scoring uses a weighted formula:

# Naive fixed-weight approach (what every other team builds)
def compute_friction(chat_turns, attempts, score_pct, session_minutes):
    return (chat_turns/10 * 0.35 +      # 35% weight — a guess
            (attempts-1)/4  * 0.30 +    # 30% weight — a guess
            (1-score_pct/100) * 0.25 +  # 25% weight — a guess
            session_minutes/45 * 0.10)  # 10% weight — a guess

These weights are assumptions. They are wrong for every cohort in a different way:

Cohort Type	Why Fixed Weights Fail
Cohort of junior developers	Chat turns are noise — they ask everything regardless of confusion. Time-on-task matters more.
Cohort of self-taught learners	Retry attempts are the strongest signal. They never ask for help even when stuck.
Cohort in an intensive bootcamp	Score gap is the most predictive signal. Fast incorrect submission is the failure pattern.
Cohort using mobile only	Session duration is unreliable — they tab out constantly. Chat + score dominate.

A system with hard-coded weights will always misfire on some cohorts. It will generate false positives, burning mentor attention on learners who are fine. It will generate false negatives, missing learners who genuinely need help. Over time, mentors stop trusting it.

2.2  The RL Solution: Contextual Multi-Armed Bandit

Instead of hard-coding weights, the system treats each weight configuration as an "arm" in a bandit problem. The bandit selects which configuration to use when computing friction. After an intervention, the outcome tells the system whether that configuration was correct. The weights evolve.

Why a Bandit and not Q-Learning or Deep RL?
•  Q-learning requires defined state transitions — we have no clear MDP here.
•  Deep RL requires thousands of episodes before convergence — impractical for a cohort of 40.
•  UCB1 Bandit works perfectly for the problem of "which parameter set is best" — exactly what we need.
•  The bandit is interpretable: judges and mentors can see the weights changing in real time as the system learns.

2.3  The UCB1 Algorithm
UCB1 (Upper Confidence Bound 1) balances exploration of untried weight configurations with exploitation of known-good ones:

# UCB1 formula for selecting which weight arm to use next
ucb_score(arm) = avg_reward(arm)
               + C * sqrt( ln(total_pulls) / pulls(arm) )

# The second term is the exploration bonus:
# Arms tried fewer times get a higher bonus, forcing exploration.
# As data accumulates, exploitation dominates and weights stabilize.

# C = 1.41 (sqrt(2)) is the standard UCB1 constant

2.4  The Reward Signal: Closing the Loop
The reward signal is what makes this genuinely RL rather than just adaptive rules. After a mentor acts on an alert, the system waits 7 days (configurable; 1 minute in demo mode) and re-computes friction for the same learner on the same task. The delta determines the reward:

Outcome After 7 Days	Reward to Bandit
Friction dropped by > 30%	+1.0  (intervention was correct and effective)
Friction dropped by 15–30%	+0.7  (meaningful improvement)
Friction dropped by 5–15%	+0.4  (small improvement)
No change (±5%)	0.0   (inconclusive)
Friction worsened 5–15%	−0.3  (possible false positive)
Friction worsened > 15%	−0.7  (wrong signal, bad intervention)
Mentor never acted	0.0   (no signal either way)

The bandit arm that was used to compute the original friction score receives this reward. Over time, configurations that correctly identify struggling learners accumulate positive rewards and get selected more often. Configurations that generate false positives accumulate negative rewards and get deprioritized.

Different cohorts converge on different weight profiles. The system learns what predicts failure for each specific group of learners. That is something no fixed-weight formula can ever do.

The bandit explores all 16 arms early on, accumulates reward data, and converges toward the arm that is most predictive of true struggle for the specific cohort. Convergence typically occurs within 30 to 50 interventions.

 
4. How It Works End-to-End


4.1  Signal Extraction Pipeline
For each learner on each task, the system extracts four raw signals from existing tables and normalizes them to 0.0–1.0:

Signal	Computation
Chat Signal	Number of AI conversation turns / 10. Boosted by +0.20 if >40% of user messages contain struggle language. Boosted by +0.15 if AI escalated to providing solution.
Attempt Signal	Number of retries normalized over 5 attempts. Direct indicator of repeated failure.
Score Signal	1.0 minus normalized score percentage. Lower score = higher signal. Falls back to attempt count if no scorecard exists.
Time Signal	Session duration divided by 3x expected task duration. Captures time anomalies without penalizing thorough learners.

4.2  The Friction Score Formula

# Bandit selects the arm for this computation
arm = bandit.select_arm()   # UCB1 selects from 16 candidate weight sets

# Weighted composite using bandit-selected weights
friction = (arm.w_chat     * sig_chat     +
            arm.w_attempts * sig_attempts +
            arm.w_score    * sig_score    +
            arm.w_time     * sig_time)

# Severity classification
# friction >= 0.65  ->  HIGH
# friction >= 0.40  ->  MEDIUM
# friction  < 0.40  ->  LOW

# Every computation is logged to friction_computations table
# This is the bandit's experience replay buffer

4.3  Individual vs Systemic Classifier
This is the core intellectual differentiator of the entire system. For each task, the classifier aggregates friction scores across all learners in the cohort and determines whether the problem is an individual issue or a content issue:

def classify_cohort_signal(task_id, all_learner_friction_results):
    struggling_pct = count(f for f in results if f.severity in HIGH/MEDIUM)
                     / total_learners

    if struggling_pct >= 0.35:   # 35%+ of cohort struggling
        return {
            'signal_type': 'SYSTEMIC',
            'owner': 'CREATOR',
            'message': 'Content problem. 40% of cohort struggling.',
            'action': 'regenerate_task_with_ai'
        }
    else:
        return {
            'signal_type': 'INDIVIDUAL',
            'owner': 'MENTOR',
            'message': 'N learners struggling. Content is fine for the rest.',
            'action': 'mentor_outreach'
        }

The classifier also detects common weak rubric dimensions by aggregating question_scorecards across all learners. A rubric dimension with cohort-average below 60% is flagged as a systemic skill gap, independent of the overall friction classification.

4.4  The Reward Loop
After an intervention is logged, the background reward tracker fires after 7 days. It re-computes friction for the learner on the same task using current data, compares to the pre-intervention friction, and awards a reward between -0.7 and +1.0 to the bandit arm that was used for the original computation. The bandit’s UCB scores update, and weight selection improves for the next computation.

Full Feedback Loop:
1.  Bandit selects arm  →  compute friction  →  alert fires  →  mentor acts  →  intervention logged
2.  7 days later: re-compute friction  →  measure delta  →  compute reward
3.  Reward credited to the arm that made the original prediction
4.  UCB scores update  →  better arm gets higher selection probability next time

 
5. New API Routes


5.1  GET /intelligence/mentor/{cohort_id}/briefing
Returns top 5 priority learners with friction scores, driving signals, confidence levels, and the bandit’s current weight configuration. This is the mentor’s daily pre-read. Reuses existing get_cohort_completion(), get_cohort_streaks(), and chat_history queries.

Response Field	Description
priority_alerts	Count of learners requiring immediate attention
learners[]	Top 5 learners ranked by priority score (friction × confidence)
cohort_health	AT_RISK / MONITOR / HEALTHY with average friction
bandit	Current arm, total learning rounds, best arm so far, exploration phase flag

5.2  GET /intelligence/cohort/{cohort_id}/anomalies
Aggregates friction across all learners for every task in the cohort. Runs the Individual vs Systemic classifier per task. Returns two lists: systemic issues (for creators) and individual issues (for mentors), sorted by severity.

5.3  GET /intelligence/bandit/{cohort_id}/state
Returns full bandit learning state including all 16 arms with pull counts, average rewards, and UCB scores. Also returns weight history trend for the visualization chart. This is the endpoint that powers the live RL demo moment.

5.4  POST /intelligence/actions/mentor-intervention
Generates an AI-powered, evidence-specific outreach message using gpt-4.1-mini and logs the intervention to the database with a link to the friction_computations record. The link enables reward attribution 7 days later. Background task schedules the reward check.

5.5  POST /intelligence/bandit/{cohort_id}/process-rewards
Processes pending rewards for all interventions whose 7-day window has elapsed. Updates bandit UCB scores, saves weight history snapshot. Can be called manually for demo mode with a 1-minute window.

5.6  GET /intelligence/creator/{cohort_id}/rubric-heatmap
Aggregates question_scorecards across all learners and tasks to build a criterion-by-task matrix. Returns the heatmap data plus global weakness analysis. Powers the most visually striking element of the demo.

 
6. Production-Grade Features


6.1  Mentor Pre-Read
Before a weekly cohort review, the mentor sees a 5-bullet briefing per learner: what they struggled with, what improved, what to ask about, and a ready-to-send outreach message. Not a dashboard. A workflow tool that slots into something mentors already do.

6.2  Individual vs Systemic Split Screen
The visual showstopper of the demo. Left panel shows an individual learner’s friction signals. Right panel shows the cohort-wide heatmap for the same task. Judges understand the product’s value in 2 seconds from this single view.

6.3  Live Bandit Weight Visualization
An animated bar chart showing the 16 weight arms with their pull counts and average rewards. A button simulates 30 historical interventions, causing the bars to animate as the system learns. This is the demo moment that no other team has. Judges see reinforcement learning happening in real time.

6.4  Rubric Weakness Heatmap
A color-coded matrix with tasks as rows and rubric criteria as columns. Each cell shows the cohort-average score for that criterion on that task. Red cells indicate systemic skill gaps. A global weakness analysis identifies criteria that are consistently low across multiple tasks. One-click triggers context-aware AI regeneration for red tasks.

6.5  Context-Aware Content Regeneration
When the system triggers a task regeneration job, it injects the full signal context into the existing task_generation_jobs infrastructure: which rubric dimensions are weak, what percentage of learners struggled, what confusion topics emerged from chat history. The AI regenerates with evidence, not blindly.

6.6  Signal Confidence Scoring
Every friction alert carries a confidence level computed from data completeness: number of chat turns, attempt count, score availability, and days in cohort. LOW confidence alerts are shown but not escalated. HIGH confidence alerts trigger full intervention workflows. This is honest AI — showing uncertainty rather than hiding it.

6.7  Completion Velocity Decay Detection
Per-learner rolling 7-day completion rate is compared to their personal baseline from the previous 7 days. A drop of more than 50% triggers a velocity deceleration flag on the learner’s priority card. This catches burnout and disengagement at Day 5, not Day 30.

6.8  Mentor Inaction Escalation
If a HIGH severity learner has been flagged to a mentor for more than 48 hours with no action taken, the interventions table triggers an admin-level escalation. This creates mentor accountability without requiring manual tracking.

11. Final Verdict


The One Killer Feature
The UCB1 Bandit Weight Learner + Individual vs Systemic Classifier, combined.
Not because it is the most impressive algorithm. Because it answers the only question that matters operationally: Is this learner’s problem unique to them, or is it the content’s fault?
